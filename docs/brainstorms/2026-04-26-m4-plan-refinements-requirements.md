# M4 Plan Refinements

## Problem Frame

The M4 plan (`docs/plans/2026-04-26-001-feat-m4-streaming-summary-and-cost-controls-plan.md`) was structured well after the initial draft + scope/feasibility/coherence review pass, but a subsequent ideation pass against three frames (adversarial / failure modes; scope-and-simplicity cuts; cross-cutting M5/M6 + UX leverage) surfaced 7 survivors after dedupe. Two of the seven are silent-correctness bugs that would defeat M4's headline goals. Two are cross-cutting hooks that cost ~5-80 lines in M4 and unblock M5/M6. The remaining three are scope hygiene.

This document captures the durable decisions from the brainstorm so the M4 plan can be revised in one targeted pass without re-litigating the choices.

The 7 ideation survivors map to refinements R1-R4, R6-R8 below (R5 was deferred to M5/M6 during the document-review pass — its slot is preserved in the numbering for traceability against the ideation doc, but it carries no M4 work). Survivor #3 (streaming-cancellation cleanup contract) was further subdivided into R3a-R3d during brainstorm because each sub-path has its own snapshot/ledger discriminator. The rejection summary in the ideation doc is the disposition record for the other 25 candidates.

**Terminology used throughout this doc:** `BrainSnapshot` (capitalised) refers to the Python dataclass at `apps/agent/archmentor_agent/brain/decision.py`; `brain_snapshots` row refers to the persisted Postgres record at `apps/api/archmentor_api/models/brain_snapshot.py`. "Compaction" is the action; "compactor" is the agent component that performs it.

See seed: `docs/ideation/2026-04-26-m4-plan-review-ideation.md`.
See precedent: `docs/brainstorms/2026-04-25-m3-plan-refinements-requirements.md` (M3 used the same ideation → refinements-bundle → plan revision flow).

## Requirements

**Silent-correctness fixes (must address — currently broken in the plan as written)**

- **R1.** *Fingerprint stability across Haiku compaction.* Unit 1's brain-input fingerprint must remain stable when the Haiku compactor's CAS apply appends to `session_summary` on a parallel path without changing the brain-relevant input slice. **Drop `summary_chars` from the fingerprint inputs entirely.** Hash `(transcript_turn_count, decisions_count, phase, active_argument.topic, event_payload)` only — no `transcript_window_hash`, no `summary_chars` bucketing. Compaction's transcript-window decrement IS captured by `transcript_turn_count` and SHOULD flip the fingerprint when it occurs (compaction genuinely changes brain-input — the brain now reads a compressed summary plus a smaller window). What the narrowing prevents is summary-mutation alone (which the brain's prompt builder does include via `session_summary`, but which doesn't materially change the brain's decision surface) flipping the fingerprint. **Property test invariant (binding):** fingerprint stable across `session_summary` mutation that does NOT change `transcript_turn_count`. The previously-considered bucketed-`summary_chars` projection was rejected during the plan-review pass because realistic Haiku appends are 300–700 chars per compaction — straddling any bucket size below ~2000 chars and defeating the bucket's job. Without this fingerprint-input narrowing, the cost throttle silently degrades to no-op whenever the compactor appends to `session_summary` between two otherwise-identical brain inputs. Verified against plan Unit 1 §305 and Unit 5 §R8 / R9.

- **R2.** *PHASE_TIMER bypasses the cooldown gate.* Unit 1's exponential-backoff `_cooldown_until_ms` resets only on TURN_END. PHASE_TIMER fires every 30 s during over-budget phases and exists *to break a stuck silence*. The cooldown rule must be: PHASE_TIMER bypasses the cooldown gate. Without this, a `consecutive_stay_silent=4` cooldown of 32 s eats the very PHASE_TIMER nudge designed to break the silence. PHASE_TIMER still passes through the fingerprint-skip gate, but the binding correctness requirement is the cooldown bypass; per-phase 90 s nudge dedup (Unit 7 §R13) is the operational dedup mechanism since `over_budget_pct` increments monotonically with elapsed time and back-to-back identical fingerprints are rare in practice. Verified against plan Unit 1 §307 (cooldown matrix) and Unit 7 §R13.

- **R3.** *Streaming-cancellation cleanup contract.* Unit 3 / Unit 4 must specify behaviour for four mid-stream paths the current draft leaves implicit:
  - **R3a.** On `TimeoutError` or external `CancelledError` arriving mid-stream, call `tts_stream.aclose()` (the livekit-agents 1.5.4 `SynthesizeStream` cancel-and-wait + channel-close primitive at `livekit/agents/tts/tts.py:673`). Because `KokoroEngine.synthesize` is atomic and the daemon thread keeps emitting audio frames for up to 2 s after `aclose()` returns (per `apps/agent/archmentor_agent/tts/kokoro.py:55-61, 138-143`), R27 dispatch must NOT fire until the trailing audio drains. **Behavioural requirement (binding):** R27 dispatch awaits a drain-complete signal from the streaming TTS path with a wall-clock fallback at Kokoro's documented `join(timeout=2.0)` floor + a small margin (~0.5 s). R3b's "any partial played → suppress R27" check runs *after* the wait completes so the suppression decision sees the true "did the candidate hear partial audio" state. **Implementation note (deferred to plan Unit 4):** today no drain-complete signal is exposed from `_stream_engine` — `consumer_gone` is the consumer→worker stop signal, not the worker→done signal. Plan Unit 4 picks the mechanism (a new asyncio Event signalled cross-thread from the drain thread's exit points; the framework's `wait_for_idle()`; or another primitive) — the brainstorm only fixes the behaviour. Without this synchronization, R27 races the still-draining frames or fires after a half-sentence dangles in silence.
  - **R3b.** When any partial `utterance` audio has played, suppress R27's "Let me come back to that — please continue." but still set `_apology_used=True` so R27 doesn't re-attempt. The half-sentence the candidate already heard plus R24's elapsed-time copy carries the visible signal; layering R27 on top produces audible double-talk.
  - **R3c.** On cancellation post-utterance pre-`message_stop` (the brain finished streaming `utterance` but cancellation arrived before the final `message_stop` event), construct a new `BrainDecision.partial(utterance=accumulated, reason="cancelled_mid_stream")` factory in `apps/agent/archmentor_agent/brain/decision.py` and write a snapshot row carrying the accumulated utterance string + a `partial=True` flag. The factory is explicit (not `stay_silent` with a side-channel `utterance` field) so M5/M6 readers fail loudly on the unknown discriminator instead of silently misclassifying the row as "abstention" (the candidate did hear audio). The factory reuses the existing `_silent` skeleton at `decision.py:181-205` but populates `utterance` instead of forcing it to None; `decision.decision="stay_silent"` and `decision.reason="cancelled_mid_stream"` are the discriminators. Replay tools that don't recognise the new reason raise on the unknown discriminator — the right loud-failure mode.
  - **R3d.** On post-stream schema-violation with partial TTS already playing, push the canned closing token *"— actually, let me think again."* through the same `tts_stream` before `end_input`, and ledger the row as `reason=schema_violation_partial_recovery`. Always closes the half-sentence on a complete clause; interviewer-voice tail (mirrors M3 R27/R28). Without a tail, the next dispatch may steelman an unrelated point and the candidate hears a jarring jump.

**Cross-cutting M5/M6 leverage**

- **R4.** *`SessionTelemetry` in-memory aggregator + structured session-end log line.* Build a `SessionTelemetry` dataclass on `MentorAgent` that maintains a tightly-scoped set of running counters across the session, each tied to an M4 dogfood-gate observable: `ttfa_ms_histogram: list[int]` (raw samples; p50/p95/max computed at digest time — backs Unit 10 "TTFA < 2 s on >80% of turns"), `brain_calls_made: int` (cost-rate denominator), `skipped_idempotent_count: int` and `skipped_cooldown_count: int` (cost-throttle effectiveness — backs the `--cost-throttle-stats` replay flag), `dropped_stale_count: int` (backs Unit 10 "zero `queue.dropped_stale` lines" gate), `compactions_run: int` (compaction trigger frequency vs the 30-turn threshold). Six fields total. On session end (router shutdown drain, before ledger client close), emit one structured `agent.session.telemetry` log line carrying the dataclass as a single JSON field. **No Postgres column, no Alembic migration in M4.** Dropped from earlier draft (deferred to whichever future milestone has a real consumer): `compactions_failed`, `phase_nudges_fired`, `argument_rounds_max`, `interruptions_made`, `phase_durations_s_actual` — none of which back an M4 dogfood-gate observable; surviving as M4 fields would re-introduce the same speculative scope the column-deferral cut. **Implementation hooks needed (deferred to planner):** each field maps to a specific increment site in router/agent code; the planner enumerates the hooks during plan revision.

- **R5.** *Deferred to M5/M6 (out of M4 scope).* The original survivor proposed a `prompt_content_hash` column on `brain_snapshots` to enable M6 eval-harness ghost-diff joins. Per the plan-review pass: M3-era rows are already NULL on this column either way, M6's join semantics are unresolved, and shipping the column in M4 only saves one M5/M6 migration that the same milestone is going to write anyway. **R5 is dropped from the M4 refinements bundle.** When M6 designs its eval harness, it adds whichever schema element (column, payload field, separate table) actually fits the query. Until then, `brain_snapshots` stays as M3 left it, and the seven other refinements (R1, R2, R3a-d, R4 in-memory form, R6, R7, R8) are the M4 bundle.

**Scope hygiene**

- **R6.** *Inline `summary_compaction_threshold` as a module constant; keep `haiku_model` as a `Settings` field.* Plan Unit 5 currently adds two new `Settings` fields. Cut only one: inline `summary_compaction_threshold = 30` as a module constant in `apps/agent/archmentor_agent/brain/haiku_client.py` next to `HaikuClient` (mirrors `_BRAIN_DEADLINE_S = 180.0` in `brain/client.py`), documented with the rationale "30 turns ≈ 2-3 minutes of conversation under nominal cadence; raise for slower-paced phases observed during M4 dogfood." Keep `haiku_model` as `Settings.brain_haiku_model: str = "anthropic/claude-haiku-4-5"` because the model id is operator-tuned for the same gateway-routing reason `Settings.brain_model` already lives in `Settings` (see `apps/agent/archmentor_agent/config.py:96-102`) — direct Anthropic vs Unbound deployments need different prefixes. `BRAIN_RATES` (`brain/pricing.py`) retains both `claude-haiku-4-5` and `anthropic/claude-haiku-4-5` rows so a snapshot replays under either gateway. If post-dogfood operator tuning becomes needed for the threshold, wire through `Settings` then — this is the same discipline CLAUDE.md applies to `_MIN_SPEECH_RMS`.

- **R7.** *Cut the `"keep"` schema sentinel; switch consumer to key-presence test.* Revert the brain output schema to `{type: ["object", "null"]}` (no string union). The brain emits an object to set/replace, an explicit `null` to close, and omits the key for "no change." Distinguishing absent-key from explicit-`null` requires a consumer-side change: `apps/agent/archmentor_agent/state/session_state.py:182-184` currently uses `dict.get("new_active_argument") is not None` which collapses absent-and-null into one path. Switch to `"new_active_argument" in updates` (key-presence test). With that consumer change, `_resolve_active_argument` becomes: object → set/replace per topic-equality rules; explicit null → clear (with `argument_resolved=true` ledger discriminator); absent → preserve prior unchanged; auto-clear-stale (3 min, rounds=0) handles forgotten openers. M2/M3 era replay snapshots stay deterministic because the absent-key rule matches their original semantics. **Rationale (corrected from earlier draft):** the sentinel was redundant — `with_state_updates` already collapses absent and explicit-`None` to no-change at the value level; the sentinel added a third schema-violation surface without adding any guarantee. Auto-clear-stale only covers the `rounds==0` case; for `rounds>=1` mid-argument, "preserve" is the correct semantics and both schemas produce it. Drop the planned union schema, the planned brain-prompt instruction "emit `\"keep\"` to leave unchanged," and the planned backwards-compat shim.

- **R8.** *Drop the four `docs/solutions/*.md` writeup deliverables.* Plan Units 1, 3, 4, 5 each list a `docs/solutions/<topic>.md` writeup as a unit deliverable. None are load-bearing for M4 acceptance: no dogfood gate checks them, no test references them, no production code imports from them. M3 plan flagged the `docs/solutions/` gap and didn't fill it; M4 punting it again is honest. If post-dogfood a real lesson hardens, write the writeup then, against actual production code rather than against the plan's mental model.

## Key Decisions

- **Decision (R3 partial-recovery tail):** Push *"— actually, let me think again."* in interviewer voice on schema-violation-with-partial-TTS. Rationale: matches M3 R27/R28 voice; always closes the half-sentence on a complete clause; ~0.5 s of audio cost; ledger discriminator (`reason=schema_violation_partial_recovery`) lets eval harness filter these from candidate-utterance metrics. Considered: neutral filler ("— hmm.") rejected as too short to absorb prosody jitter; no-tail rejected because the half-sentence dangling produces "Walk me through your capacit— [pause] — How would you partition?" jarring jumps.
- **Decision (R3a synchronization):** R27 dispatch awaits a drain-complete signal from the streaming TTS path with a wall-clock fallback at the kokoro `join(timeout=2.0)` floor + ~0.5 s margin. Rationale: `KokoroEngine.synthesize` is atomic; cancellation can't actually stop in-flight audio (CLAUDE.md gotcha + `kokoro.py:55-61, 138-143`). Without explicit synchronization, R27 either races the trailing audio (audible double-talk) or fires after silence (R3b suppresses R27 prematurely). The mechanism (new asyncio Event signalled cross-thread vs framework primitive vs polling) is plan-Unit-4 territory; the brainstorm pins behaviour, not implementation choice.
- **Decision (R3c factory):** Add a new `BrainDecision.partial(utterance, reason)` factory at `apps/agent/archmentor_agent/brain/decision.py` that reuses the `_silent` skeleton but populates `utterance` instead of forcing it to None. Rationale: explicit factory means M5/M6 readers fail loudly on the unknown discriminator; reusing `stay_silent` with a side-channel utterance silently misclassifies the row as "abstention" when the candidate did hear audio. The discriminator path is `decision.decision="stay_silent" + decision.reason="cancelled_mid_stream" + decision.utterance is not None` — replay tools that check `reason` get the right answer; tools that only check `decision` see "stay_silent" and need to know they're missing partial-utterance rows.
- **Decision (R4 transport):** In-memory `SessionTelemetry` dataclass + one structured `agent.session.telemetry` log line on session end. **No Postgres column, no Alembic migration in M4.** Rationale (per plan-review pass): M4's dogfood gate is log-based and doesn't query Postgres for telemetry; M5/M6 add their own column or ledger event when their query patterns are designed; a JSONB column on `sessions` would commit the schema for unwritten consumers (CLAUDE.md "no speculative features"). Considered: JSONB column on `sessions` (rejected as speculative); ledger event (rejected as buried-row pattern that M5 would need to extract anyway).
- **Decision (R5 deferred):** Drop `prompt_content_hash` from the M4 refinements bundle. Rationale: M3-era `brain_snapshots` rows are already NULL on this column either way; M6's join semantics are unresolved; shipping the column in M4 only saves M5/M6 one migration that the same milestone would otherwise write itself. The four ideation survivors that motivated R5 (M6 ghost-diff joins) aren't unblocked by an unindexed nullable column ahead of any M6 design work.
- **Decision (R6 inline location):** `apps/agent/archmentor_agent/brain/haiku_client.py` (next to `HaikuClient`), mirroring `_BRAIN_DEADLINE_S` in `brain/client.py`. Rationale: policy constants live with their consumer. Only `summary_compaction_threshold` is inlined; `haiku_model` stays in `Settings` per the next decision.
- **Decision (R6 `Settings.brain_haiku_model` retained):** The `claude-haiku-4-5` and `anthropic/claude-haiku-4-5` model-id rows are operator-tuned (which gateway you route through), so the model id stays parameterised as `Settings.brain_haiku_model: str = "anthropic/claude-haiku-4-5"`. Mirrors the existing `Settings.brain_model` (`apps/agent/archmentor_agent/config.py:96-102`). `BRAIN_RATES` retains both prefixed and bare ids so a snapshot replays correctly under either gateway.
- **Decision (R7 schema + consumer wiring):** Cut the `"keep"` sentinel; brain emits object/null/absent (Option A from pass-1 review). Consumer at `apps/agent/archmentor_agent/state/session_state.py:182-184` switches from `dict.get("new_active_argument") is not None` to `"new_active_argument" in updates` (key-presence test) so absent-key vs explicit-`null` are distinguishable. Rationale: explicit-null is the brain's only clean way to close an argument; without it, brain is stuck "preserve forever" until auto-clear-stale fires (which only fires at `rounds==0` after 3 min). The consumer change is the load-bearing edit; the schema simplification is a downstream consequence.
- **Decision (R1 fingerprint inputs):** Drop `summary_chars` from the fingerprint entirely; hash `(transcript_turn_count, decisions_count, phase, active_argument.topic, event_payload)` only. Rationale (from plan-review pass): bucketed projection over `summary_chars` was unstable because realistic Haiku appends (300-700 chars per compaction) straddle any bucket size below ~2000 chars. The brain doesn't make different decisions because the summary grew 400 chars; it makes different decisions because transcript count, decisions, phase, or argument topic changed — those four are the brain-relevant signals.

The planner enumerates the hooks during plan revision; this brainstorm specifies the dataclass shape, not the wiring.
- *[Affects R7][Technical]* Whether `with_state_updates` should accept a `keys_present: set[str]` sidecar from the router so the absent-vs-explicit-null distinction is recoverable for non-`active_argument` future state-update fields.
